LLM & Prompt

先别急着谈 RAG，先知道模型的边界

只有搞清楚 LLM 的训练路径、能力来源和固有局限，后面才知道为什么单靠提示词不够，为什么超长上下文只是过渡，为什么 RAG 会成为更现实的工程方案。
预训练 + 微调 泛化能力 + 涌现能力 提示词工程不是万能药

AI 能力链路

  • 人工智能 → 机器学习 → 深度学习 → LLM → 生成式 AI
  • 预训练：无标注大数据学习语言规律
  • 微调 SFT：用指令或标签数据定向强化能力

模型为什么强

  • 泛化：能迁移到没见过的相似任务
  • 涌现：规模上来后会出现少样本学习、指令跟随、复杂推理
  • “像懂”本质是统计规律在起作用

模型为什么不够

  • 时效性差，训练完就开始过期
  • 专业知识不稳定，容易幻觉
  • 安全、隐私、合规边界天然敏感
提示词工程 · 能救场，但不能包办全局

系统提示词

负责定边界、定角色、定输出格式，像系统里的“隐藏规则”。

用户提示词

负责提问题、补背景、给约束，决定本轮对话怎么走。

有效提示词习惯

结构化表达、少歧义、正向提问、一步一步思考、少量样本 ICL。

超长上下文

优点是简单直接，缺点是不可扩展、受窗口限制、成本高，不适合长期做底座。

📚
Why RAG

RAG 不是炫技，是在给回答找依据

RAG 的核心不是让模型突然变聪明，而是让模型在回答前先“看资料”。所以它解决的不是语言能力，而是真实性、时效性、专业性和私有知识可用性。
先检索，再生成 降低幻觉 补足领域知识
为什么只靠模型不够

只靠 LLM

  • 回答速度可以很快，但“依据”不稳定
  • 训练语料过期后，时效性问题会直接暴露
  • 进入垂直领域后，越专业越容易胡说

接入 RAG

  • 先查知识库，再让模型组织语言
  • 私有数据、实时知识、专业文档能进入回答链路
  • 回答不只是流畅，还要可追溯、可验证

RAG 的一句话版本：用户提问后，系统先在知识库里找相关内容，再把这些内容拼进提示词，最后交给 LLM 生成回答。也就是 `问题 → 检索 → 提示词构建 → LLM → 回答`。

解决幻觉

让回答优先依赖外部资料，而不是只靠模型“猜”。

解决时效

知识库更新比重训模型快得多，更适合业务迭代。

解决专业性

垂直行业文档能直接接入，减少通用模型的偏差。

解决私有知识

企业内部信息可以通过检索进入回答，而不是公开训练。

Real Case

现实案例：CloudSync 的客服知识助手

CloudSync 是一家 200 人规模的 SaaS 公司。Support 团队每天要处理大量产品问答、版本说明、权限配置和历史工单， 但知识分散在帮助中心、发布公告、内部 SOP 和聊天记录里，导致响应慢、口径乱、新人上手难。
📦 业务场景：客服与工单协同 👥 核心角色：Support / PM / 算法 / 后端 🎯 目标：更快、更准、更一致

为什么这个案例值得放进来：它不是一个纯技术实验，而是一个非常典型的企业知识型场景。用户问的问题看起来像一句自然语言，但背后可能同时牵涉版本、权限、套餐、功能说明和历史处理规则。如果只靠模型记忆，答案会很快过期；如果只靠关键词检索，又很难理解真实意图，所以必须把完整框架放进去跑。

首响越来越慢

帮助中心、版本公告、SOP 和历史工单分散在不同系统里，Support 每次都要来回翻资料，首响时间从 4 小时拉到 11 小时。

回复口径不一致

同一个问题，不同同事给出的解释和处理建议不一样，客户会怀疑平台到底有没有统一标准。

新版本一发布就过期

功能名称、入口位置、套餐规则一变，旧知识马上失效，单靠模型记忆根本跟不上。

新人培训成本高

新同事知道文档在哪，不代表知道该看哪一段，更不代表能把零散信息组合成一句可回复的话。

把这套框架塞进这个项目，会发生什么

只靠 LLM 不够

模型可以把话说顺，但遇到版本差异、套餐约束和内部规则时，回答会越来越像“听起来对”，不一定真的对。

A-F 要做什么

先把帮助中心、版本公告、内部 SOP、历史工单和处理规则接成知识库底座，保证资料干净、可解析、可分块、可检索。

G-H 要做什么

用户说“客户开不了单”，系统不能直接搜字面，而要先判断是权限、套餐、网络还是版本差异，再拆成可检索的子问题。

最终交付什么

不是一个聊天框，而是一套带知识更新、检索增强、回答兜底、人工转接和效果监控的业务链路。

这类项目真正盯的不是“会不会回答”，而是这 4 个结果

业务目标

  • 把首响时间拉回目标线
  • 提高回复一致性，减少反复转接
  • 缩短新人独立处理问题的时间
  • 让版本更新后的知识同步更快

必须同步设计的机制

  • 知识边界清单和数据源优先级
  • 版本更新后的知识同步 SOP
  • badcase 收集、重排和回归验证机制
  • 回答不确定时的人转人兜底链路

Offline Build

知识库构建 A-F：RAG 的底座在离线阶段

这里开始进入真正的工程落地。重点不是背 A-F 名字，而是知道每一步的目标、风险、质量标准，以及 AI 产品经理要盯哪些关键决策。
数据采集 文档解析 清洗 / 分块 / 向量化 / 索引

原始数据采集

先定义知识边界、确认数据来源、量级和优先级，搞清楚什么该接、什么不能接。

产品经理职责：基于业务场景定义知识边界，协调业务方、数据方和法务一起确定可用源。

文档解析

把 PDF、Word、网页、表格等材料转成可处理文本。重点不只是“能解析”，而是“解析后是否能用”。

产品经理职责：定义文档解析质量标准，判断关键格式是否完整保真。

数据清洗

做冗余剔除、格式统一、敏感信息过滤、异常处理，保证数据真正适合进入检索系统。

产品经理职责：制定清洗规则，平衡自动化与抽检，不让脏数据污染答案。

文本分割

核心原则是语义完整、长度适配、保留必要重叠。切太碎丢上下文，切太大检索又不准。

产品经理职责：定义 chunk 策略，优先按自然语义边界切，而不是机械按字数硬切。

文本向量化

把文本变成 embedding，让系统能按语义相似度找内容。模型是否适配行业术语非常关键。

产品经理职责：关注向量模型和业务语义的适配度，不是只看“大家都用什么”。

向量索引构建

让大规模向量能快速近似检索。常见有树状、哈希、图、聚类、关键词等方案。

产品经理职责：根据速度、精度、数据量、部署成本做选择，理解向量库与关系库分工。

这部分最容易被忽略的结论：很多人以为 RAG 效果差是“模型不行”，但实际问题常常发生在离线阶段。数据边界没定义清楚、解析质量差、清洗规则松、chunk 切得乱、embedding 模型不适配，都会让后面答案出问题。

Online Retrieval

在线阶段 G-H：用户问一句，系统背后要做很多事

真正上线后，用户给出的往往不是标准问题，而是含糊问题、跨领域问题、多轮追问问题。所以在线阶段的关键，不是“查一下”，而是先理解问题，再组织更聪明的检索路径。
Query 改写 多路查询 / 分解 重排与阈值验证

Pipeline G：Query 改写

  • 识别意图类型、领域细分类、多轮追问关系
  • 做语义强化、歧义消解、上下文补全、意图拆解
  • 典型例子：“Java 怎么泡”要先判断是语言还是咖啡

Pipeline H：多路查询

  • 把复杂问题拆成多个可检索子问题
  • 做多路召回、重排和阈值选择
  • 最终不是分数越高越好，而是和真实场景适配度更高
Query 增强 6 种方法

语义强化

补充同义表达和关键业务词，让查询更完整。

歧义消解

解决多义词、代词不明、领域切换问题，避免检索跑偏。

上下文补全

把多轮对话里的隐含信息补回这一轮 query。

意图拆解

复杂问题先拆层级，再分别检索，不让单次召回负担过重。

术语适配

把行业黑话、缩写、别名映射到系统能理解的标准表达。

约束强化

把时间、地点、品牌、型号、身份等约束补得更明确。

Optimization

检索优化、生成优化，还有更高级的 RAG 形态

到了这里，重点开始从“能跑通”升级成“能跑得稳、跑得久”。RAG 不是“查一下文档然后扔给模型”这么简单，真正好用的系统，检索和生成都要做分层优化，复杂问题还要拆、补、跳、校验。
结构化检索 vs 语义检索 生成环节 5 步 Graph RAG / Speculative RAG
检索系统不是一件事，是两件事

结构化检索

  • 按字段、条件、规则、SQL 精确筛选
  • 适合岗位、城市、价格、型号这类硬约束
  • 强在明确，不强在语义理解

语义 / 混合检索

  • 按 embedding 找相近内容
  • 再配合关键词召回、重排提升准确性
  • 适合问答和非结构化知识场景

RAG 降成本思路

把 chunk 设计成多级结构，先用二级片段粗召回，再映射回更完整的一级片段。这样既能减少 token，又能保住语义完整度。

生成环节 5 个流程

提示词构建、LLM 生成执行、内容优化与后处理、多轮上下文管理、复杂问题处理策略，缺一块都容易出短板。

复杂问题处理

问题分解、层次化推理、多跳检索，先拿宏观，再补细节，不靠一次检索硬撑到底。

Graph RAG

不是只查相似文本，而是查实体关系和结构化关联。更适合股权穿透、政策合规、科研追踪、根因分析。

Speculative RAG

让更专门的 RAG 草稿器并行产出多个候选答案，再由通用模型评估接收，适合知识密集型问题。

这两者给你的启发

RAG 正在从“先检索再生成”走向更复杂的协同架构，未来重点会是多结构知识和多阶段推理。
