CHAPTER 01

模型为什么需要 Prompt

模型会生成，但不会自动知道你要什么。
图解：输入不是文字，而是一条预测链
文字人类自然语言
Token切成片段序列
预测继续猜下一个
输出再拼成答案

预测机器

定义：LLM 本质是 next-token prediction。
重要性：它不会自动理解你的真实业务目标。

例：同样是“退款”，不加上下文时可能答政策，也可能答流程。

训练三阶段

预训练：学语言和世界知识。
SFT：学会按指令回答。
RLHF：更贴近人类偏好。

作用：把“能生成”一步步训练到“更会按要求生成”。

Prompt 作用

发生环节：模型调用时。
作用：设目标、加约束、控格式。

例：要求“先给结论，再引用规则，再给下一步建议”。

CHAPTER 02

三种 Prompt 各管一层

规则归规则，输入归输入，输出起点单独管。
图解：Prompt 优先级堆栈
System规则、角色、边界
最稳的一层
User当次问题和意图
Pre-fill锁住开头和格式

System

定义：长期规则层

作用：控制角色、边界、口吻。
角色设定 回复口吻 风险边界
常放：长期稳定规则，例如“只能依据官方规则回答”。
别放：用户这次临时问题，否则每轮都要重写一遍。
例：你是电商客服，只能依据平台售后政策回复用户。

User

定义：当次输入层

作用：说明本轮问题和补充条件。
用户诉求 订单状态 临时变量
常放：本轮问题、业务变量、用户补充信息。
别放：长期固定规则，这些应该提前沉到 System。
例：我的订单已签收，退款多久到账？订单号是 12345。

Pre-fill

定义：输出起跑线

作用：锁 JSON、标题或列表格式。
JSON 骨架 固定标题 列表开头
常放：输出模板、结构骨架、固定开头。
别放：复杂业务规则，Pre-fill 更适合控形式，不适合管策略。
例：请按“结论 / 依据 / 下一步”输出，首句先给明确结论。
图解：一轮调用到底怎么拼
稳定规则放 System，变化信息放 User，固定输出骨架放 Pre-fill。拆对了，模型才稳定。
System 你是售后客服，只能依据平台规则答复，语气清晰克制。
User 订单已签收，用户问“退款多久到账”，并补充了订单号。
Pre-fill 请按“结论 / 依据 / 下一步”三段输出，先给直接答案。
结果 回复既稳又像产品定义的样子，不容易跑偏。

CHAPTER 03

好 Prompt 不是灵感，是结构

先搭积木，再用样例和调试把它磨顺。
图解：六要素拼装台
角色你是谁例：你是售后客服助手。
任务做什么例：判断用户诉求并给回复。
背景补信息例：订单已签收 3 天，支持原路退款。
示例给样板例：参考一条优秀客服回复作为风格样本。
约束划边界例：不能承诺超权限补偿，不能编造规则。
格式定输出例：按“结论 / 依据 / 下一步”三段输出。
图解：调试不是重写，是单变量循环
写初版先跑起来
找问题一次只看一处
改一处别全盘重写
再测试留版本记录

CHAPTER 04

先判象限，再选策略

先判断“谁知道”，再决定“怎么问”。
我知不知道
AI知不知道

双方都知道

直接用，但要验。

例：总结一条公开的退款政策。

AI知道 我不知道

拿来探索，但别全信。

例：先让它解释“向量检索”这类陌生概念，再自行核对。

我知道 AI不知道

主动喂业务背景。

例：把你们公司的 SLA、规则、黑名单条件贴进去。

双方都不知道

接工具，别瞎编。

例：预测新策略效果时，先接数据、搜索或实验，而不是让模型硬猜。

CHAPTER 05

Context 不是一句话，是整个舞台

看见什么、先看什么、能看多久，都会影响回答。
图解：Context 四层
专业一点说，Context 是模型在当前推理时刻能够看到的全部信息集合。
System固定规则层，优先级最高例：回答必须先给结论，再引用平台规则。
History对话记忆层，负责承接上下轮例：上一轮用户已经说过“订单已签收”。
Retrieved外部知识层，通常由 RAG 注入例：召回“退款到账 T+1 到 T+3”的规则片段。
User当前任务层，驱动本轮输出例：用户本轮真正问的是“退款多久到账”。
图解：位置即权重
开头规则最稳
中间最易被淹没
结尾最新最显眼
四种失效
污染错误信息往后扩散
稀释重点被噪音淹没
溢出超窗后前文失忆
漂移慢慢偏离原约束

CHAPTER 06

System Prompt 要像规格，不像随手文案

不是一句“你是助手”，而是一份可上线的规格。
图解：一个合格的 System Prompt 应该包含什么
System Prompt 更像长期控制规格，负责把产品要求稳定地翻译成模型规则。
角色你是谁例：你是售后客服助手，不是万能顾问。
能力你会什么例：可解释规则、提炼诉求、给下一步建议。
约束你别做什么例：不能编造订单状态，不能承诺额外赔付。
格式怎么输出例：固定输出“结论 / 依据 / 动作建议”。
兜底不会时怎么办例：信息不足时先追问订单号或转人工。
要分模块
要有版本和测试
要能和工程对接

CHAPTER 07

PM 的 Prompt 工作流是一条产品流水线

不是最后补一句，而是从验证到迭代都要管。
图解：Prompt 在产品全周期里出现在哪里
D1-D2
需求 先试可行性 产出物：问题清单、demo prompt、可行性判断。
D3-D4
设计 同步写草稿 产出物：System Prompt v0.1、变量表、few-shot 样例。
D5-D7
开发 对齐注入位 产出物：Prompt 拼装逻辑、Context 注入位、接口字段约定。
D8-D9
测试 做四类验收 产出物：正常 / 边界 / 对抗 / 降级用例，评测结果表。
持续
上线后 看真实对话再改 产出物：坏案例库、版本日志、优化 backlog。
最后记 3 件事
会写 Prompt 还不够
还得会管 Context
最后要做成团队资产
