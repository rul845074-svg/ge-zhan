# 个站2全流程图B — 开发规格文档

> 本文档记录 `个站2全流程图B.html` 的设计决策、组件结构、数据格式与待办事项。
> 续写章节只需在 `CHAPTERS` 数组里补充数据，不需要改 HTML/CSS/JS 结构。

---

## 一、项目背景与设计方向

### 1.1 为什么新建这个文件

原版 `个站1全流程图.html` 存在以下问题：
- 文字密集（大段对话气泡），重点被淹没
- CSS 头像丑，漫画感弱
- 内心OS、旁白、对话并列，信息层次混乱
- 看完一章不知道「这章PM做了什么、学到了什么」

### 1.2 设计方向（B方案：流程节点卡片式）

放弃密集分镜格子，改为：
- **六章 = 六个流程节点**，顶部时间轴串联
- **每章 = 一张结构化卡片**，分区呈现关键信息
- **故事感靠视觉传达**：emoji场景图 + 情绪emoji + 一句话，不靠大段文字
- **内心OS作为彩蛋**：点故事线格子触发，不占主界面空间
- **面向面试官**：每章突出「挑战—解法—产出物—PM能力」

---

## 二、技术架构

- **单 HTML 文件**，CSS/JS 全内联，本地打开即用
- **数据驱动**：所有章节内容存于 `CHAPTERS` 数组，JS 负责渲染，不手写 DOM
- **字体**：ZCOOL KuaiLe（标题）+ Noto Sans SC（正文），Google Fonts CDN
- **响应式**：桌面优先，移动端降为单列

## 四、章节数据结构（CHAPTERS 数组）

每个章节对象的完整字段说明：

```js
{
  id: 1,                          // 章节编号（1-6）
  emoji: "🔥",                    // 章节标志 emoji
  shortName: "信号汇聚",           // 短标题（时间轴显示）
  stage: "产品机会发掘",            // 中文阶段名（卡片副标题）
  stageEn: "Product Discovery",   // 英文阶段名（卡片标题区水印）
  processTerms: [                 // 专业流程术语标签（显示在章节头部）
    "Opportunity Assessment",
    "Signal Triangulation"
  ],
  color: "#E8622A",               // 主题色（banner、时间轴激活态、标签等）
  soft: "#FFF0E6",                // 浅色版（tag背景）
  border: "#FFCFB0",              // 边框色（tag边框）
  tagline: "从三路散乱信号里，识别出同一个产品机会假设",  // 一句话定位
  scene: "办公室各角落 → 产品VP办公室",   // 场景描述
  time: "项目前史 · 一切开始之前",        // 时间节点
  difficulty: 1,                  // 难度 1-3（渲染为星星）
  diffLabel: "向上沟通 · 机会判断",      // 难度说明

  // 故事线格子（每格 = 一个关键节点，建议4-6格）
  panels: [
    {
      mood: "warm",               // 格子氛围色：warm/conflict/insight/win/normal
      icon: "📊",                 // 场景图标（大，顶部）
      who: "Support负责人",       // 角色职位（必须与 roles[].role 一致才能触发OS）
      moodEmoji: "😤",            // 情绪emoji（代替内心OS文字）
      line: "首响时间 4h → 11h，人根本不够用"  // 一句话核心台词（≤20字）
    }
    // ...
  ],

  // 核心信息（info-grid四格）
  challenge: "...",               // 核心挑战（2-3句）
  solution: "...",                // 解决路径（2-3句，含专业方法论名称）
  capabilities: ["信号捕捉", "竞品监测", "机会假设构建", "向上汇报与资源申请"],

  // 角色内心OS（彩蛋数据）
  // role 必须与 panels[].who 保持一致，点格子才能触发
  roles: [
    {
      role: "PM",
      emoji: "🧑‍💻",
      os: ["OS文案1（随机选一条）", "OS文案2"]
    }
    // ...
  ],

  // 会议信息
  meetings: [
    {
      title: "产品VP 1on1 · 机会汇报",   // 会议名称
      type: "向上汇报",                  // 会议类型
      attendees: ["PM", "产品VP"],       // 参会职位（不写名字）
      duration: "15 min",
      note: "以三信号框架汇报机会，争取调研授权"  // 可选备注
    }
  ],
  decision: "获准开展一周深度调研，进入可行性评估阶段（Feasibility Study）",  // 本章关键决策

  // 产出物
  deliverables: [
    {
      icon: "📄",
      name: "产品机会分析 Brief",
      desc: "一页纸；含三信号来源、问题假设、竞品参照、初步ROI预估区间"
    }
    // ...
  ],

  // 踩坑反思
  pitfall: "...",    // 踩坑点（1-2句）
  lesson: "...",     // 反思/正确做法（1-2句）

  // 下一章悬念（渲染在 next-bar 里）
  suspense: "..."    // 最后一章（id:6）不需要此字段
}
```

## 五、交互说明

### 5.1 章节切换
- 点击顶部时间轴节点 → `showChapter(id)` → 重新渲染卡片 + 平滑滚动到卡片顶部
- URL hash 同步（`#ch1` ~ `#ch6`），刷新后恢复当前章节
- 点击 next-bar（下一章悬念条）也可切换

### 5.2 内心OS彩蛋
- 故事线格子底部显示「点击看内心OS 💭」提示（仅有OS数据的格子显示）
- 点击格子 → `showOSFromPanel(event, role, quotes)` → 从该角色的 `os[]` 随机取一条
- 弹出气泡：`position: fixed`，定位于格子正下方，4.5秒后自动消失，点空白处也关闭
- **坑**：弹窗用 fixed 定位，top值不能加 `window.scrollY`（已修复）

### 5.3 会议模块职位chip
- 参与角色职位也可点击触发OS弹窗（与格子共用同一套弹窗逻辑）

## 八、本次对话沉淀

### 8.1 关键决策记录

| 决策点 | 结论 | 原因 |
|--------|------|------|
| 是否在原文件改 | 新建文件，不动原版 | 改造幅度大，保留原版作为备份 |
| 漫画格子 vs 流程卡片 | 选B：流程节点卡片式 | 原版文字太多、重点淹没，卡片式更利于面试官扫读 |
| 角色用名字还是职位 | 职位（PM/产品VP/Support负责人…） | 泛化性更强，面试场景更专业 |
| OS彩蛋放哪里 | 故事线格子点击触发 | 会议模块单调，加互动有趣；OS不占主界面空间 |
| 产出物详细程度 | 带说明（name + desc两行） | 只写名字太单薄，desc说清楚"含什么"更有说服力 |

### 8.2 踩过的坑

1. **OS弹窗不显示**：popup 用 `position:fixed` 但 top 值加了 `window.scrollY`，导致页面一滚动弹窗就飞到视口外。Fixed 定位用视口坐标，去掉 scrollY 即可。

### 8.3 用户偏好（本次确认）

- 工程量大时，先做框架+第一章，确认方向再继续
- 参与角色只写职位，不写角色名字
- 少文字多视觉，emoji承载情绪而非文字OS
- 产出物要详细说明内容构成
- 专业流程术语（英文）要在显著位置体现
